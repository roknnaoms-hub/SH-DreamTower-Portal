# SH드림타워 지식산업센터 정보 웹포털 시각화 방안

## 1. 목적
- SH드림타워 방문자/입주사/관리자가 층별 입주기업 정보를 빠르게 탐색할 수 있는 웹포털 구축
- "층 선택 → 입주기업 선택 → 상세정보 확인" 흐름을 2~3클릭 내에 완료하도록 UX 단순화
- 향후 공실 관리, 민원/공지, 방문 예약 기능으로 확장 가능한 정보 구조 설계

## 2. 사용자 시나리오
1. 사용자가 포털 메인에서 건물 전체 층 목록을 본다.
2. 특정 층(예: 10F)을 클릭한다.
3. 해당 층 입주기업 리스트가 나타난다.
4. 기업 카드(호실/상호)를 클릭하면 상세정보 패널이 열린다.
5. 상세정보(업종, 연락처, 웹사이트, 영업시간, 위치)를 확인한다.

## 3. 정보 구조(IA)
- 건물
- 층(Floor)
- 호실(Unit)
- 입주기업(Tenant)
- 공지/이벤트(Optional)

권장 관계 모델
- 1개 층 : N개 호실
- 1개 호실 : 0~1개 입주기업
- 1개 입주기업 : 1개 이상 태그(업종, 키워드)

## 4. 화면 구성안
### A. 메인(층 탐색 화면)
- 좌측: 층 빠른 이동 패널(B4F~14F)
- 중앙: 선택 층의 호실 배치 또는 리스트
- 우측: 선택 기업 상세정보 패널(기본은 안내 메시지)

### B. 기업 상세 패널(클릭 후)
- 기본 정보: 기업명, 호실, 층
- 연락처: 대표전화, 이메일
- 소개: 2~4줄 요약
- 부가 정보: 홈페이지, 업종 태그, 입주일
- CTA: 길찾기/문의(향후 확장)

### C. 관리자 화면(2단계)
- 엑셀 업로드(층/호실/기업 정보 일괄 반영)
- 변경 이력 관리(수정자, 수정시각)

## 5. 시각화 및 인터랙션 원칙
- 클릭 가능한 요소를 명확히 하기 위해 층 버튼/기업 카드를 컬러 대비로 분리
- 현재 선택 상태를 고정 강조(Active 상태)
- 모바일에서는 3단 레이아웃을 세로 스택(층 목록 → 기업 리스트 → 상세 패널)으로 재배치
- 검색 필터 제공: 기업명, 업종, 호실

## 6. 데이터 설계(예시)
### Floor
- id (number)
- code (string, 예: "10F")
- order (number)

### Tenant
- id (number)
- floorCode (string)
- unit (string, 예: "1006")
- name (string)
- category (string)
- phone (string)
- email (string)
- website (string)
- description (string)
- status (enum: active, vacant)

## 7. API 설계(권장)
- GET /api/floors : 층 목록 조회
- GET /api/floors/{floorCode}/tenants : 특정 층 입주기업 조회
- GET /api/tenants/{id} : 기업 상세 조회
- GET /api/tenants?keyword=&category= : 검색 조회
- POST /api/admin/tenants/import : 엑셀 업로드(관리자)

## 8. 기술 스택 제안
- 프론트엔드: React 또는 Vue + TypeScript
- 백엔드: Node.js(Express/NestJS) 또는 Spring Boot
- DB: PostgreSQL
- 캐시/검색: Redis(선택), PostgreSQL FTS 또는 OpenSearch(확장 시)
- 배포: Azure App Service + Azure Database for PostgreSQL

## 9. 구현 단계(4주)
1주차
- 요구사항 확정, 데이터 표준 정의(층/호실/기업)
- UI 와이어프레임 확정

2주차
- 층/기업 조회 API 구현
- 메인 화면(층 선택, 기업 리스트, 상세 패널) 구현

3주차
- 검색/필터 및 모바일 반응형 구현
- 관리자 입력(수동 등록 또는 CSV 업로드) 구현

4주차
- QA, 성능 최적화, 운영 배포
- 관리자 가이드/사용자 안내서 작성

## 10. 운영 및 품질 지표
- 정보 조회 평균 클릭 수: 3회 이하
- 첫 화면 로딩 시간: 2초 이내
- 데이터 최신성: 수정 후 5분 이내 반영
- 오류율: 월간 API 에러율 0.5% 이하

## 11. 리스크 및 대응
- 리스크: 기존 안내판 이미지 OCR 정확도 이슈
- 대응: 초기에는 관리자가 검수한 기준 데이터셋 사용, OCR은 보조 기능으로 적용

- 리스크: 층/호실 표기 기준 불일치
- 대응: 표준 코드 룰(예: 10F-1006) 수립 및 입력 검증 적용

## 12. 제출 결론
본 방안은 "층 중심 탐색"과 "기업 상세 정보 즉시 확인"을 핵심으로 설계되어,
방문자 편의성과 관리 운영 효율을 동시에 높일 수 있습니다.
초기 버전은 빠른 조회 UX에 집중하고, 2단계에서 검색 고도화/관리자동기화를 확장하는 전략을 권장합니다.
